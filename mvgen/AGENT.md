# MVGen - Agent Development Guide

This document provides comprehensive information for AI agents (like Claude) working with MVGen codebases.

## What is MVGen?

MVGen is a code generation tool that transforms declarative service specifications (MVEP format) into production-ready code. It's a metaprogramming tool that generates:
- Protocol Buffer 3 definitions
- Go implementations
- CLI tools with command handlers
- Type-safe API interfaces

## Core Concepts

### MVEP Specifications

MVEP (MainVec Endpoint) specifications are JSON/JSONC files that describe services:

```json
{
  "$id": "service-id",
  "$schema": "https://spec.mainvec.com/mvepspec/0.1/schema/2023-09-19",
  "name": "service",
  "namespace": "namespace",
  "gen_options": {
    "go_package": "import/path;pkgname",
    "go_api_package": "import/path/api;api"
  },
  "commands": { /* command definitions */ },
  "recordsDefs": { /* record definitions */ }
}
```

### Commands

Commands are operations with input fields and result fields. Each command generates:
- A protobuf message type (e.g., `CreateUserCmd`)
- A result message type (e.g., `CreateUserCmdResult`)
- A command handler type (e.g., `type CreateUserCmdHandler func(...)`)
- CLI command setup with flag parsing
- Implementation stub function

### Records

Records are reusable data structures referenced via `recRef` type. They map to protobuf messages and Go structs.

### Field Types

| Type | Protobuf | Go | Notes |
|------|----------|-----|-------|
| `string` | `string` | `string` | |
| `boolean` | `bool` | `bool` | |
| `int32` | `int32` | `int32` | |
| `int64` | `int64` | `int64` | |
| `recRef` | message | struct pointer | Reference to recordsDefs |
| `map` | `map<string, T>` | `map[string]T` | Key is always string |

## Generated File Structure

When MVGen processes a spec file named `myservice.jsonc`:

```
output/
├── api/
│   ├── myservice.proto           # Protobuf definition (generated)
│   ├── myservice.pb.go           # Protobuf Go code (protoc)
│   └── myservice_package.go      # Handlers & dispatcher (mvgen)
├── cmd/
│   └── myservice/
│       └── myservice_main_cmd.go # CLI main (mvgen)
├── myservice_impl.go             # Implementation stubs (mvgen)
└── myservice_commands.go         # Command runner factory (mvgen)
```

### File Categories

**DO NOT MODIFY (will be overwritten):**
- `api/*.proto` - Generated from spec
- `api/*.pb.go` - Generated by protoc
- `api/*_package.go` - Generated by mvgen
- `cmd/*_main_cmd.go` - Generated by mvgen
- `*_commands.go` - Generated by mvgen

**CAN MODIFY (but may be regenerated):**
- `*_impl.go` - Implementation functions

**PROTECTED (add `// NOMVGEN` or `// NOWOGEN`):**
- Any file with protection marker at the top

## Key Generated Patterns

### 1. Handler Types (`*_package.go`)

```go
type CreateUserCmdHandler func(context.Context, *CreateUserCmd) (*CreateUserCmdResult, error)
```

### 2. Command Runner Struct (`*_package.go`)

```go
type PkgCommandRunner struct {
    RunCreateUserCmd CreateUserCmdHandler
    RunListUsersCmd  ListUsersCmdHandler
    // ... one for each command
}
```

### 3. Command Dispatcher (`*_package.go`)

```go
func (r *PkgCommandRunner) RunCmd(ctx context.Context, cmd any) (any, error) {
    switch v := cmd.(type) {
    case *CreateUserCmd:
        return r.RunCreateUserCmd(ctx, v)
    case *ListUsersCmd:
        return r.RunListUsersCmd(ctx, v)
    default:
        return nil, errors.New("unknown command")
    }
}
```

### 4. Implementation Stubs (`*_impl.go`)

```go
func runCreateUserCmd(ctx context.Context, cmd *api.CreateUserCmd) (*api.CreateUserCmdResult, error) {
    return nil, errors.New("command not implemented")
}
```

### 5. CLI Command Setup (`*_main_cmd.go`)

```go
func prepareCreateUserCmd(rootCmd *cli.Command) {
    cmd := &api.CreateUserCmd{}
    cliCmd := &cli.Command{
        Usage: "create_user",
        Short: "create a new user",
        Run: func(ctx *cli.Context, args []string) {
            result, err := commandRunner.RunCreateUserCmd(ctx, cmd)
            // handle result
        },
    }
    cliCmd.Flags().StringVar(&cmd.Name, "name", "", "user name")
    cliCmd.Flags().StringVar(&cmd.Email, "email", "", "user email")
    rootCmd.AddCommand(cliCmd)
}
```

## Working with MVGen Projects

### Identifying MVGen Projects

Look for:
1. `spec/` directory with `.json` or `.jsonc` files using MVEP schema
2. Generated files with header: `// code generated by mvgen. Should not modify`
3. `mvpapi/` directory structure
4. Dependencies on `github.com/mainvec/mvp/mvpgo` or `github.com/mainvec/ugo`

### Common Task Patterns

#### Adding a New Command

1. Edit the spec file (e.g., `spec/myservice-spec.json`)
2. Add command definition:
```json
{
  "commands": {
    "NewCommandCmd": {
      "title": "description",
      "alias": "new_command",
      "fields": { /* input fields */ },
      "resultFields": { /* output fields */ }
    }
  }
}
```
3. Regenerate: `mvgen generate --in spec/myservice-spec.json --lang go --outdir .`
4. Implement the handler in `*_impl.go`

#### Adding a Field to Existing Command

1. Edit spec file, add field with unique `fnum`:
```json
{
  "fields": {
    "newField": {
      "fnum": 5,  // must be unique and not previously used
      "type": "string",
      "title": "new field description"
    }
  }
}
```
2. Regenerate code
3. Update implementation to use new field

#### Creating a Record Type

1. Add to `recordsDefs`:
```json
{
  "recordsDefs": {
    "Address": {
      "fields": {
        "street": {"fnum": 1, "type": "string"},
        "city": {"fnum": 2, "type": "string"},
        "zipCode": {"fnum": 3, "type": "string"}
      }
    }
  }
}
```
2. Reference in commands:
```json
{
  "fields": {
    "address": {
      "fnum": 3,
      "type": "recRef",
      "recRefVal": "Address"
    }
  }
}
```

#### Implementing a Command

Edit `*_impl.go`:

```go
func runCreateUserCmd(ctx context.Context, cmd *api.CreateUserCmd) (*api.CreateUserCmdResult, error) {
    // Validate input
    if cmd.Name == "" {
        return nil, errors.New("name is required")
    }

    // Business logic
    userId := generateUserId()
    err := saveUser(ctx, userId, cmd.Name, cmd.Email)
    if err != nil {
        return nil, fmt.Errorf("failed to save user: %w", err)
    }

    // Return result
    return &api.CreateUserCmdResult{
        UserId: userId,
    }, nil
}
```

### Debugging Generated Code

#### Finding Command Handler

1. Open `*_package.go`
2. Search for `type {CommandName}CmdHandler`
3. Find corresponding implementation in `*_impl.go`

#### Tracing CLI to Implementation

1. CLI entry point: `cmd/*/___*_main_cmd.go`
2. Find `prepare{CommandName}Cmd()` function
3. Traces to `commandRunner.Run{CommandName}Cmd()`
4. Which calls implementation in `*_impl.go`

#### Understanding Protobuf Messages

1. Check `api/*.proto` for message definitions
2. See `api/*.pb.go` for generated Go structs
3. Messages follow pattern: `{CommandName}Cmd` and `{CommandName}CmdResult`

## MVGen Codebase Architecture

### Core Files

| File | Purpose | Key Functions |
|------|---------|---------------|
| `mvgen.go` | Core structures, validation | `BuildSrvDefFromString()`, `ValidateAgainstSchema()` |
| `mvgen_pb3.go` | Protobuf generation | `BuildProtoBuffDefFromSrvDef()` |
| `mvgen_go.go` | Go code generation | `GenerateGOProtoBuffAPIFromProto()`, `GenerateFromEmbeddTemplate()` |
| `mvgen_runner.go` | Command execution | `ExecuteGenerate()`, `ExecuteValidateCmd()` |

### Data Structures

```go
type SrvDef struct {
    ID         string
    Schema     string
    Name       string
    Namespace  string
    Version    string
    GenOptions map[string]any
    Commands   *omap.OrderedMap[string, *CommandDef]
    RecordDefs *omap.OrderedMap[string, *RecordDef]
}

type CommandDef struct {
    Title        string
    Alias        string
    Fields       *omap.OrderedMap[string, *FieldDef]
    ResultFields *omap.OrderedMap[string, *FieldDef]
}

type RecordDef struct {
    Fields *omap.OrderedMap[string, *FieldDef]
}

type FieldDef struct {
    Fnum       int32
    Type       FieldDataType
    Title      string
    Repeated   bool
    RecRefVal  string  // for recRef type
    ValueType  string  // for map type
}
```

### Template System

Templates are located in `resources/codegen_templates/go/`:

| Template | Generates |
|----------|-----------|
| `go_cli_main.txt` | CLI entry point |
| `go_impl_code.txt` | Implementation stubs |
| `go_package_code.txt` | Package handlers & dispatcher |
| `go_commands_runner_code.txt` | Command runner factory |
| `go_api_code.txt` | API interface definitions |

Templates use Go `text/template` with custom functions:
- `{{.Name | ToUpper}}` - Uppercase
- `{{.Name | ToTitle}}` - Title case
- `{{.Name | ToCamel}}` - Camel case

## Best Practices for Agents

### When Modifying MVGen Projects

1. **Always read the spec file first** - Understand the current commands and structure
2. **Check for existing implementations** - Don't recreate existing functionality
3. **Use consistent field numbering** - Never reuse field numbers
4. **Validate before generating** - Run `mvgen validate` first
5. **Preserve manual code** - Check for `NOMVGEN`/`NOWOGEN` markers

### When Creating New Services

1. Start with `mvgen init`
2. Define core records first
3. Design commands with clear input/output
4. Use meaningful aliases for CLI
5. Generate and test incrementally

### When Debugging

1. Check spec validation first
2. Look at protobuf definition (`.proto`)
3. Trace from CLI through package handlers to implementation
4. Verify field numbers match between spec and code

### Field Number Management

**Critical:** Field numbers must be:
- Unique within a command/record
- Never changed once assigned
- Never reused (even if field is removed)
- Sequential for clarity (not required but recommended)

## Common Pitfalls

### 1. Modifying Generated Files

**Problem:** Changes to `*_package.go` or `*_main_cmd.go` are lost on regeneration.

**Solution:** Only modify `*_impl.go` or use `NOMVGEN` marker.

### 2. Field Number Conflicts

**Problem:** Duplicate `fnum` values cause protobuf compilation errors.

**Solution:** Ensure all field numbers are unique within their scope.

### 3. Missing Record References

**Problem:** Using `recRef` to non-existent record.

**Solution:** Verify record exists in `recordsDefs`.

### 4. Case Sensitivity

**Problem:** Command names, record names are case-sensitive.

**Solution:** Use exact names as defined in spec.

### 5. Forgetting to Regenerate

**Problem:** Spec changes don't appear in code.

**Solution:** Always run `mvgen generate` after spec modifications.

### MVGen Commands

```bash
# Generate code
mvgen generate --in spec.json --lang go --outdir ./out

# Validate spec
mvgen validate --in spec.json

# Initialize new spec
mvgen init --name myservice --ns myns
```
